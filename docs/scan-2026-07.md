# How ready is the MCP ecosystem for the 2026-07-28 spec?

**We probed every remote server in the official MCP registry — all ‹TK: funnel.uniqueUrls› of them — and measured how many already clear the 2026-07-28 spec's required behavior.**

> **Nothing switches off on July 28.** The 2026-07-28 date is when the spec *text* publishes — not a flip that breaks anything. Version negotiation continues, and features deprecated in this release keep working for at least 12 months. ## What we measured

`mcp-spec-check` runs eight black-box checks. Only three can mark a server **not-ready** — they're the mandatory behavior changes:

- **`discover`** — `server/discover` replaces the `initialize` handshake in the stateless core; servers must implement it.
- **`routing-headers`** — `Mcp-Method` / `Mcp-Name` are required on every request and servers must reject header/body mismatches, so gateways can route without reading the body.
- **`session-independence`** — the protocol-level session (`Mcp-Session-Id`) is removed; session-pinned servers can't serve the stateless core behind a load balancer.

The other five (`error-codes`, `cache-metadata`, `mrtr`, `deprecated-features`, `auth-metadata`) are optional or forward-looking. They inform an **adoption matrix**, never the ready/not-ready verdict.

Two deliberate choices about honesty:

- **Auth walls are not failures.** A 401/403 means "needs credentials," not "broken." We classify those servers as auth-walled and don't grade them — except `auth-metadata`, whose RFC 9728 document is served *outside* the wall (see below).
- **We publish what we couldn't assess.** When a server answers the 2026-07-28 probes too ambiguously to judge (e.g. it answers the legacy `initialize` but rejects every new-style request with `-32600`), the check is **inconclusive** — counted separately, never guessed into a pass or fail. The ecosystem-wide inconclusive rate is ‹TK: quality.inconclusiveRatePctByCheck (summarize)›.

## Methodology

- **Scope:** every remote server in the official MCP registry (`registry.modelcontextprotocol.io`), latest active version of each, deduped by normalized URL. Not the whole ecosystem — see limitations.
- **Probing:** pure black-box HTTP. ~16–20 requests per open server, a named `mcp-spec-check-scan/‹TK: meta.toolVersion›` User-Agent, 10s per request, host-serial (one request at a time per host) so no server — including the large gateways — ever sees parallel probes.
- **Every percentage carries its denominator** (see the funnel and each section).
- **Reproducible:** the aggregate is committed alongside this post (`docs/scan-2026-07.aggregates.json`) and regenerable with `npm run scan`.

## Limitations

- **Registry ≠ ecosystem.** Servers not in the official registry aren't counted; other directories (Glama, PulseMCP) are out of scope for this snapshot.
- **Point-in-time.** A single scan on ‹TK: meta.date›; servers migrate continuously.
- **Black-box.** We observe wire behavior, not source. A server can be mid-migration in ways a probe can't see — which is exactly why ambiguous answers are reported as *inconclusive*, not guessed.
