# How ready is the MCP ecosystem for the 2026-07-28 spec?

**We probed every remote server in the official MCP registry, all 7,850 of them, to measure how many already implement the 2026-07-28 spec's required behavior. As of 2026-07-12: almost none do yet, and that's exactly what you'd expect.**

> **Nothing switches off on July 28.** The 2026-07-28 date is when the spec text publishes, not a flip that breaks anything. Version negotiation continues, and features deprecated in this release keep working for at least 12 months. Everything below is an adoption snapshot: how far the ecosystem has moved toward the stateless core, not a count of things that break. A server that isn't "ready" today keeps working today.

## The headline

Of the 4,356 servers we could probe openly, just 1 (0.02%) passes all three required checks today. 3,956 (90.8%) don't yet, and 399 (9.2%) answered too ambiguously to judge. This isn't a failure of the ecosystem. The spec text isn't even GA, and only SDK betas exist, so this is the baseline: the migration to the stateless core has barely begun, and this is the "before" picture.

The registry is heavily concentrated. The single largest host accounts for 15.5% of all target URLs, so we report every headline number twice: once per endpoint, and once host-collapsed (each host counted once, by majority vote). Collapsed, not-ready is 2,408 of 2,741 (87.9%), close to the endpoint-level 90.8%, so the headline is not an artifact of one big gateway.

## What we measured

`mcp-spec-check` runs eight black-box checks. Only three can mark a server **not-ready**, and those are the mandatory behavior changes:

- **`discover`**: `server/discover` replaces the `initialize` handshake in the stateless core; servers must implement it.
- **`routing-headers`**: `Mcp-Method` / `Mcp-Name` are required on every request and servers must reject header/body mismatches, so gateways can route without reading the body.
- **`session-independence`**: the protocol-level session (`Mcp-Session-Id`) is removed; session-pinned servers cannot serve the stateless core behind a load balancer.

The other five (`error-codes`, `cache-metadata`, `mrtr`, `deprecated-features`, `auth-metadata`) are optional or forward-looking. They inform the adoption matrix below, never the ready/not-ready verdict.

Two deliberate choices about honesty:

- **Auth walls are not failures.** A 401/403 means "needs credentials," not "broken." We classify those servers as auth-walled and don't grade them, except `auth-metadata`, whose RFC 9728 document is served outside the wall (see below).
- **We publish what we couldn't assess.** When a server answers the 2026-07-28 probes too ambiguously to judge (for example, it answers the legacy `initialize` but rejects new-style requests with a non-standard error code), the check is inconclusive: counted separately, never guessed into a pass or fail.

## Findings

### Readiness

| view | ready | not-ready | unknown |
| --- | ---: | ---: | ---: |
| endpoint-level (of 4,356 open) | 1 (0.02%) | 3,956 (90.8%) | 399 (9.2%) |
| host-collapsed (of 2,741 hosts) | 1 (0.0%) | 2,408 (87.9%) | 332 (12.1%) |

The 9.2% "unknown" are servers where no required check outright failed but at least one couldn't be decided. The bulk of the 90.8% not-ready is driven by `routing-headers` (3,893 of 4,356 open servers don't validate the new routing header) and `discover` (2,773 don't implement `server/discover`, returning method-not-found or equivalent).

### Newest protocol version demonstrated

The centerpiece: for each open server, the newest version it will demonstrably speak, meaning it did something only a 2026-07-28 server does (`server/discover` advertised the version, or it enforced the new routing header, or it declared the version), not merely that it happened to be stateless. (An earlier cut of this analysis over-counted 2026-07-28 by roughly 500x precisely because 2,511 old servers, some negotiating 2024-11-05, are coincidentally stateless; statelessness alone proves nothing.)

| version | endpoint-level | host-collapsed |
| --- | ---: | ---: |
| 2026-07-28 | 5 | 5 |
| 2025-11-25 | 1,922 | 1,713 |
| 2025-11-05 | 3 | 3 |
| 2025-06-18 | 381 | 349 |
| 2025-03-26 | 1,470 | 186 |
| 2024-11-05 | 498 | 408 |
| modern (version undeclared) | 14 | 14 |
| unknown | 63 | 63 |

2025-11-25 (the current release) is the modal version; a long tail still negotiates versions from 2024 and 2025. Five servers demonstrably speak 2026-07-28.

### RFC 9728 auth metadata, measured through the wall

The one signal readable for auth-walled servers, and a number nobody else has: of 2,008 auth-required servers, 1,524 (75.9%) publish OAuth Protected Resource Metadata (RFC 9728) at the well-known path. Auth-walled servers, as a group, are markedly further along on authorization hardening than the open population is on the stateless core.

### Adoption of the optional features (open servers)

| feature (check) | adopted | not yet | notes |
| --- | ---: | ---: | --- |
| `-32602` for resource-not-found (`error-codes`) | 1,635 | 121 still legacy `-32002` | 1,663 had no resources capability to probe; 936 inconclusive |
| cache metadata `ttlMs`/`cacheScope` | 6 | 3,883 | brand-new; near-zero adoption |
| `resultType` on results (`mrtr`) | 1 | 3,871 | brand-new; near-zero adoption |
| no deprecated capabilities declared | 3,943 | 278 declare one | Logging / `resources.subscribe` |

### Data quality

0 probe crashes, 1 target hit the 120s budget, 522 targets needed a retry. Per-check inconclusive rates: `discover` 36.3%, `session-independence` 31.5%, `error-codes` 21.5%, the rest at or below 11%. The two elevated rates are the same population: servers that answer the legacy `initialize` but reject the new-style probes with a non-standard error (`-32000`, `-32600`, and friends). We report these as inconclusive rather than guessing; publishing "we couldn't assess this many" is the point, not something to hide.

### Host concentration

The largest single host holds 15.5% of all target URLs; the top 10 hosts hold 23% (host identities redacted, since this is neutral registry composition with no readiness attached to any host). This is why every headline number above is also reported host-collapsed: so you can see the endpoint-level percentages are not an artifact of a few large gateways being counted thousands of times. They are not; the two views track closely.

## A server that already passes

(Hand-verified, positive-only, consistent with reporting aggregates rather than a shame-list.)

**GitHub's MCP server** (`api.githubcopilot.com/mcp`) is auth-walled and publishes RFC 9728 protected-resource metadata; probed with a token, it passes the required checks. It is a concrete existence proof that the stateless core is shippable today.

## Methodology

- **Scope:** every remote server in the official MCP registry (`registry.modelcontextprotocol.io`), latest active version of each, deduped by normalized URL. Not the whole ecosystem; see limitations.
- **Probing:** pure black-box HTTP. Roughly 16 to 20 requests per open server, a named `mcp-spec-check-scan/0.2.0` User-Agent, 10s per request, host-serial (one request at a time per host) so no server, including the large gateways, ever saw parallel probes.
- **Every percentage carries its denominator** (see the funnel and each section).
- **Reproducible:** the aggregate is committed alongside this post ([`scan-2026-07.aggregates.json`](scan-2026-07.aggregates.json)) and regenerable with `npm run scan` from the [repo](https://github.com/Roee-Tsur/mcp-spec-check).

## Limitations

- **Registry is not the ecosystem.** Servers not in the official registry aren't counted; other directories (Glama, PulseMCP) are out of scope for this snapshot.
- **Point-in-time.** A single scan on 2026-07-12; servers migrate continuously, and we expect these numbers to climb toward and past GA.
- **Black-box.** We observe wire behavior, not source. A server can be mid-migration in ways a probe can't see, which is exactly why ambiguous answers are reported as inconclusive rather than guessed.

## Check your own server

```bash
npx mcp-spec-check https://your-server.com/mcp
```

You get the same one-line verdict, `ready for 2026-07-28: YES / NO / UNKNOWN`, that this scan is built on, plus per-check detail and links to the migration docs. Reproduce the whole scan with `npm run scan`.
